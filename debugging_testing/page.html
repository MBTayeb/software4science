It turns out that a lot of time within software projects is spent on finding and fixing bugs, i.e. small, but obstructive errors in the source code. At least for larger programming projects, the testing and optimization of code becomes a task on its own.

Debugging can be (and very often is) as simple as putting an output statement at the place in your code, where things seem to go wrong. If (lots of) errors turn up upon compilation, it is often helpful to just fix the first error, as the other errors might just be follow-up ones. If you have no idea what (or actually where) the problem is, 'delta debugging' might help: put a statement in the middle of your code and see whether the problem occurs before or after. Then again 'half' the problem again and again until you narrowed it down to a few lines. On the command line, but usually more comfortable as part of GUI editors or IDEs, proper debugging programs are available, which let you step line by line through your code during execution while at the same time showing you the values of the variables involved. More complex problems can often be found more easily this way, as you spot 'weird' values for your variables ('not-a-number' etc.) early on.

Professional software testing is a business on its own, but you can easily integrate some ideas into your code development process. At the basic level, languages like Python and C/C++ (but not Fortran -- see why commercial software developers hate it?) allow you to use 'assertions' in your code, where you can make your code stop as soon as things develop in an unexpected direction. The next step up is unit testing, i.e. to write tests with known in- and output for 'units' of your code. For languages like Python and C/C++, unit testing frameworks are available to speed up this process. A good strategy is to also turn all 'complex' bugs which you find later on into tests to avoid running into them again. Another step up is integration testing, i.e. checking whether units are working together correctly and then system testing, where you can for instance include a test suite with example cases and known results as part of your code directory. (People of course have invented many more intermediate levels of testing ... ) A very interesting option is continuous integration, which means that after each git push, a whole range of tests (unit or up to system level) is run automatically. Some people advocate 'test-driven development', i.e. to first make specifications (what should the program or part do?), then write tests based on these specifications and make them properly fail, and only then start to write solutions that will not trigger the tests.
