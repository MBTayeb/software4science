Like mentioned before, containers are made from an image. Images are themselves built from a Dockerfile. In fact, a dockerfile defines an image. In other word, it is a script including all the commands needed for the image. It must start with the command FROM image:tag to specify the name of the parent image and its version. If no tag is specified, the latest version is used.
RUN command is used to run some command inside the container, e.g. install a package.
WORKDIR path specifies the path as the working directory inside the container.
USER user[:group] sets the user for the following commands and optionally the group.
COPY copies file from a path in the host system to a path in the container.
ENTRYPOINT sets the default command to run when the container starts.
More details can be found in the official documentation https://docs.docker.com/reference/dockerfile/.

docker build path is used to build an image with the Dockerfile, which should be located in the specified path, named exactly "Dockerfile" and thus without an extension. -t image:tag flag sets the name and the version of the new image.

A user is either specified with a username or a uid, which is a unique number for the user. Analogously, the group can also be specified with a gid. Non-root users have a uid and a gid that are higher or equal to 1000, such that the first user is 1000:1000. It is very important to think about which user rights are needed. Installing new packages inside the container needs the root user. However, it is not recommended to run a container with a root user, because it would have access to the host system. A good strategy would be to set the root user in the beginning of the Dockerfile to be able to install needed packages and select another user in the last lines of the Dockerfile, i.e. when root rights are not needed anymore. Moreover, one should know which user is in the parent image. If the user is root, it is not needed to set it again in the beginning of the Dockerfile. Furthermore, if the parent image specifies a non-root user, it would be practical to set the same one in the end of the Dockerfile, or else a new user should be created.

Writing a Dockerfile is not necessary, as one could use an image from the Docker Hub https://hub.docker.com/ . But this comes to expense of flexibility and customization. One should also make sure that the dockerfile does not compomise security in someway by loading things from unknown pages for example. As this might be challenging for beginners, they could just filter out containers that are not marked as trusted.

Next, we would like to make a container from the image. This is done with the command docker run image:tag.
--rm sets automatic deletion for the container once it is stopped.
--name Container_name sets a name for the container we are starting.
-it lets us interact with the container.
-d is used to run the container in the background.
To automatically run something when the container starts, one should specify it in the end of the run command, e.g. one could add /bin/bash in the end to start a bash shell from within the container.
More details can be found in the official documentation https://docs.docker.com/reference/cli/docker/container/run/.

To access JupyterLab or another web app running within the container, it is necessary to publish ports from the container to the host system. This is done with -p 10000:8888 to publish the port 8888 from the container to the port 10000 in the host system. To understand this, some background in networking is needed. An IP address is the identifier of a computer in a network. But when one wants to share data between two entities within a network, we want it to go exactly where it is needed, that is why we need ports. Ports allow the system to specify the origin and the destination of data. Since the container is isolated from the computer, they can't share information. Publishing a port binds the port in the container to another one and thus allows everybody in the network to access the service, unless a firewall forbids it. You can get more information in this video https://www.youtube.com/watch?v=g2fT-g9PX9o

To transfer data between the host system and the container, you can either copy the data or mount it. The first method, can be done in the Dockerfile before the image is built, as we have seen above, or when the container is already running, which is done with the command docker cp path_host Container_name:path_container .
Switching the order of the paths would copy the data from the container to the host system. Copying within the Dockerfile has also the advantage that the owner of the copied data can be set to the user in the container with the flag.

The second method would be to mount a path in the host system to a path in the container. It is done in the docker run command with the flag -v path_host:path_container. This means that the container would have access to the data in the specified path in the host system.

Note: Instead of a path in the host system, data can also be transferred (copied or mounted) to the container from a volume, which is a directory that is easily accessible for containers. It allows sharing and persisting data between containers in a flexible and scalable way. (Read more in https://docs.docker.com/storage/volumes/)

Last but not least, commands can be executed in the container using the host terminal with
docker exec Container_name command
