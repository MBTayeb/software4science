Leading a reproducible workflow in Research Software Engineering (RSE) requires several components. One of the most important ones is ensuring reproducible environments. While this can be achieved in any language, this discussion will focus specifically on Python.

How to set up reproducible python environments

  1. Set up your environments per project:
    Stick with dedicated virtual environments for each project. Global library installations are convenient at first, but cumbersome in the long run.
  2. Keep track of used python version, libraries and their exact versions:

    For Python several ways exist to achieve this (see below for details on the available tools). A common but limited approach to managing reproducible Python environments is to list the Python version and library dependencies in text files such as runtime.txt (for the Python version) and requirements.txt (for libraries). Alternatively, tools like Conda’s environment.yml or pipenv with a Pipfile and Pipfile.lock can be used to define and freeze the environment.

    However, these methods can be hard to maintain or may not fully capture the actual state of the environment. To solve this problem, more modern and robust alternatives have been released. The one standing out is uv, a fast Python package manager that builds and locks environments. Using uv's pyproject.toml and lockfile (uv.lock) approach can lead to more reliable and consistent setups, especially in collaborative or long-term projects.

Reproducible jupyter workflow

Another challenge to a reproducible workflow is working with Jupyter notebooks. While they are great for exploration, they can make reproducibility and version control difficult.

To improve this, some new ideas have been developed:

  • Reactive notebooks:
    After running a cell, these notebooks update automatically all cells that depend on it. This keeps notebooks consistent and up-to-date, which saves time and prevents a lot of errors.
  • The percent format:
    Jupyter notebooks include outputs of the cells and reading them as text files is far from being practical. This makes version control complicated. With the percent format, plain script files (e.g. '.py') are used like jupyter notebooks. Here, special comments like # %% are used to split code into cells. This allows a much simpler version control, Many IDEs are now supporting this. Installing jupytext makes this also available in Jupyter. Moreover, Jupytext allows to convert a Notebook file to the percent format.
