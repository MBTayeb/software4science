Compilers, software libraries, and build systems are at the heart of RSEng and we therefore have to understand them at least at a basic level.

If you are using an interpreted language, you do not need a compiler: Upon execution of your script, a program (for instance bash or python) reads your script and executes the commands. If you are using a compiled language (like Fortran or C/C++) you need a fitting compiler, which can 'translate' your source code into a machine executable (and often optimized) program, the 'binary'. A range of compilers is available both for Fortran and C/C++. In the realm of Fortran, you'll commonly find the Intel Fortran compiler and GFortran (for Linux) as typical choices. When it comes to C/C++, Microsoft Visual C++ (for Windows) and GCC (for Linux) are standard compiler options.

Compiling is usually a two-step process (although as a default, a full program is produced if just one source code file is given): First, object files are produced, then these object files are 'linked' to produce the final program. The fun thing is that upon linking one can not only link to one's own object file, but also to already available objects, which are usually collected within so-called 'software libraries'. These libraries can be linked either static (the code is really build into the binary) or dynamic (at run time, a call is made to the library installed on the machine). Static linking does not require the necessary libraries to be installed on the computer, where the binary should run, but it blows up the size of the binary and incompatibilities might arise. Dynamic linking does not only allow keeping the binary small, but the underlying libraries can also be kept up to date, for instance regarding security issues. On Linux, static libraries are named lib<name>.a, while dynamic libraries are named lib<name>.so, on Windows it's <name>.lib and <name>.dll respectively. To link to one of these libraries, you will not only have to give the name, but also the path to where the library is placed on your system.

Whenever you can, you should use existing libraries! This saves not only your time and most likely also execution time, but increases the reliability and transferability of your work. Libraries to which basically every program links are the Fortran and/or C/C++ 'run-time' libraries. Many scientific programs link to mathematical libraries like the Intel Math Kernel Library (MKL) for all sorts of mathematical operations or to computing-related libraries like those for Message Passing Interface (MPI) calls. See below for a -- necessarily incomplete -- list of interesting libraries for scientific programming.

The tool ldd allows to investigate binaries and shows, to which libraries a binary links; these must be installed to allow execution. If you give it a try you will find that even the simplest binary link to a set of 'run time' libraries which account for very basic functionality, like input/output etc. Further information on binaries can be acquired with the program nm. You can create you own libraries from compiled object files with the tool ar, which is often a good way to organize programming projects. Besides linking, compilers usually have options to adjust the level of optimization that is tried on the code and that can also be switched off completely for debugging purposes via a separate option. When developing your application, use this debugging flag, to avoid 'weird' errors or messages from optimization.

Typical libraries include ...
  • BLAS (basic linear algebra software for vector and matrix operations etc.),
    LAPACK (linear algebra package for eigenvalue problems, matrix decomposition etc),
    FFT(fast fourier transform)
    - which are all available as part of the Intel MKL
  • MPI (message passing interface)
  • LIBINT (integral routines) or LIBXC (exchange/correlation functionals) in Quantum Chemistry
  • ...

Once you have linked the library, you can simply call the functions in your code supplied by the library like call dgemm(...)in Fortran - but you first have to look up the interface, i.e. what values to give and to get for a proper call. This information can be taken from the documentation of the library which is usually available online.
