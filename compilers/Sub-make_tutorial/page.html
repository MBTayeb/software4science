GNU make is a CLI build tool, i.e. it helps to easily build and rebuild the code in your project as needed. The make commands should normally be written in a extensionless file named makefile in the project folder. Building is then done by just running the shell command make within the same folder. What differs makefile from other script files is that they check dependencies before building.

For example, let us compare between a makefile and a shell script in the following situation. We have a Fortran script that calculates a function f(x) in different x values and saves the results to a dat file. Then, we plot the results using a python script. We already run the programs and have a plot file. Now we edit the plot file to improve the plot. Using a shell script that compiles the Fortran script and runs the python file would calculate the function values again, which is actually not needed and may waste much time. Using a makefile, we can easily avoid this problem. In fact, it automatically finds out that only the python script was edited, that is, make just runs it again. This makefile may look like: 

fig.png: code.out plot.py
	./code.out
	python plot.py
code.out: code.f90
	gfortran -Wall -Wextra -o code.out code.f90
clean: 
	rm -f *.out *.png *.dat
One should note that this makefile also allows us to easily remove all files with the extensions ".out" (binaries), ".png" (pictures) and ".dat" (data files) just by running make clean. In fact, clean is here a target that no other target depends on, which means it does not happen unless it is specified in the prompt.
If you are running this on Windows, change the rm -f command to del and the ".out" extension to ".exe". Actually, linux binary files often don't have an extension at all, but we use the ".out" extension since it easier to clean.
You can find example python and Fortran files below.
code.f90 
program f
   implicit none
   real, parameter :: a = -10, b = 10
   integer, parameter :: N = 10000
   real, parameter :: step = (b - a) / (N - 1.)
   real, dimension(N) :: x, y
   integer :: i
   do i = 1, N
      x(i) = a + (i - 1.) * step
      y(i) = atan(x(i))
   end do
   open (1, file='data.dat')
   do i = 1, N
      write (1, *) x(i), y(i)
   end do
end program f
plot.py 
import matplotlib.pyplot as plt
import numpy as np
data = np.loadtxt('data.dat')
x = data[:, 0]
y = data[:, 1]
fig, ax = plt.subplots()
ax.plot(x, y)
ax.set_xlabel("x")
ax.set_ylabel("y")
ax.set_title("f(x) = arctan(x)")
ax.grid()
plt.savefig("fig.png")
plt.close()

The makefile above was used for explanation purposes. But the project makefiles you would normally see look more like the following one. 

#CONFIG_ADD = yes
FC = ifc
FC_FLAGS = -fast
OBJECTS = main.o sub.o
ifdef CONFIG_ADD
OBJECTS = $(OBJECTS) add.o
endif
my-program: $(OBJECTS)
	$(FC) $(LD_FLAGS) -o $@ $(OBJECTS)
%.o: %.f90
	$(FC) $(FC_FLAGS) -c -o $@ $<
clean:
	rm -f *.o
clean-all:
	rm -f *.o my-program
Here, the Intel Fortran compiler (ifc) is used for compilation. The compiler and the flags are specified in the beginning of the file. This makefile builds the final executable my-program, and the objects needed for it are also listed. IF CONFIG_ADD is set to yes, add.o is added to the objects.

The next makefile could be used to compile and run any simple Fortran 90 program using gfortran without any editing. 

ifndef BIN
  $(error You must specify BIN, e.g., 'make BIN=code')
endif
FC = gfortran
FC_FLAGS = -Wall -Wextra -g -O3 -fbacktrace
.PHONY: run clean
run : $(BIN)
	./$^
%: %.f90 
	$(FC) $(FC_FLAGS) -o $@ $^
clean:
	rm $(BIN)
Now, let us discuss how to compile and run a Fortran 90 file called code.f90. %.f90 is used to replace any file in the same directory as the makefile that has the extension ".f90". If such a file exists, there will be a target with the same name but without any extension. This is called a pattern rule. In our case, the target will be code.
Makefiles always start by making the prerequisites of the first target listed before making that target. Still, running make won't do anything when using only a pattern rule as it is not enough for making a target, the target should be explicitly specified. Thus, one solution would be to add a target called all and add code as a prerequisite for it. This, however, requires editing the makefile manually, preventing a truly dynamic workflow.
The solution recommended here is to define the BIN variable directly from the shell and fill it with the filename of the target binary as in make BIN=code. This way, the target is specified on the fly and recognized by the makefile since it matches the wildcard. One should also mention, that this is only possible because the task itself is not complicated, as we only have to compile a simple Fortran program. Sometimes one needs to compile multiple sources, which can't be done with this makefile. You could find help on this below in the tutorial made by NeuralNine.
The compilation command includes automatic variables, i.e. is variables that don't have to be defined beforehand. $@ is filled with the target. $^ is filled with the prerequisites.
Moreover, we use the target .PHONY to specify the phony targets, i.e the fake ones that are not actual files. Indeed, there could be a file called clean in the same directory, which would confuse the makefile. Using .Phony makes sure that this problem doesn't happen. If one uses the first solution mentioned above, all should also be added to the phony targets.
Last but not least, we substitute the value of the BIN variable by writing$(BIN) in the prerequisites of the runphony target so that the binary is executed with ./$^. To prevent using the makefile without defining this variable, a condition was added in the beginning.
Note:
  • We don't need to add run to the shell command, since it is the first target of the makefile.
  • Because of the used condition, we can only run the clean command if we define the BIN variable.
  • If we put the run command under the pattern rule too and the leave the run target with no commands, we would still get a result once. After that, there is nothing to be done for make, as it would only run again if the fortran file is edited and thus needs to be compiled.
  • We chose to give the binary file no extension here, since the variable lets us specify it under the clean target.
