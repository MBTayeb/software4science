Compilers, software libraries, and build systems are at the heart of RSEng and we therefore have to understand them at least at a basic level.

If you are using an interpreted language, you do not need a compiler: Upon execution of your script, a program (for instance bash or python) reads your script and executes the commands. If you are using a compiled language (like Fortran or C/C++) you need a fitting compiler, which can 'translate' your source code into a machine executable (and often optimized) program, the 'binary'. A range of compilers is available both for Fortran and C/C++. In the realm of Fortran, you'll commonly find the Intel Fortran compiler and GCC (for Linux) as typical choices. When it comes to C/C++, Microsoft Visual C++ (for Windows) and GCC (for Linux) are standard compiler options.

Compiling is usally a two-step process (although as a default, a full program is produced if just one source code file is given): First, object files are produced, then these object files are 'linked' to produce the final program. The fun thing is that upon linking one can not only link to one's own object file, but also to already available objects, which are usually collected within so-called 'software libraries'. These libraries can be linked either static (the code is really build into the binary) or dynamic (at run time, a call is made to the library installed on the machine). Static linking does not require the necessary libraries to be installed on the computer, where the binary should run, but it blows up the size of the binary and incompatibilities might arise. Dynamic linking does not only allow keeping the binary small, but the underlying libraries can also be kept up to date, for instance regarding security issues. On Linux, static libraries are named lib<name>.a, while dynamic libraries are named lib<name>.so, on Windows it's <name>.lib and <name>.dll - to link to one of these libraries, you will not only have to give the name, but also the path to where the library is placed on your system.

Whenever you can, you should use existing libraries! This saves not only your time and most likely also execution time, but increases the reliability and transferability of your work. Libraries to which basically every program links are the Fortran and/or C/C++ 'run-time' libraries. Many scientific programs link to mathematical libraries like the Intel Math Kernel Library (MKL) for all sorts of mathematical operations or to computing-related libraries like those for Message Passing Interface (MPI) calls. See below for an -- necessarily incomplete -- list of interesting libraries for scientific programming.

The tool ldd allows to investigate binaries and shows, to which libraries a binary links; these must be installed to allow execution. If you give it a try you will find that even the simplest binary link to a set of 'run time' libraries which account for very basic functionality, like input/output etc. Further information on binaries can be acquired with the program nm. You can create you own libraries from compiled object files with the tool ar, which is often a good way to organize programming projects. Besides linking, compilers usually have options to adjust the level of optimization that is tried on the code and that can also be switched off completely for debugging purposes via a separate option. When developing your application, use this debugging flag, to avoid 'weird' errors or messages from optimization.

Once your projects has grown to more than a handful of source code files (to make your project accessible to anyone but yourself, each source code file should not contain more than a few hundred lines of code), the process of compiling your program can become very tedious, especially if you have different compilers, options, variables etc. for compilation on different systems. This is were build systems like make come into play; they allow you to specify rules for handling files and thereby to automate the process of compilation. Once you download a code that makes proper use of make, you just have to change into the source code directory and type make to do all necessary compilation steps. The makefile itself can be generated on the target system, using system specific variables and options. This is why most projects require you to do the 'classic' three step procedure to compile and install a new program: './configure && make && make install' (The first one generates the specific makefile, the second does the compilation and the third command copies the produced binaries to a pre-defined installation directory like $HOME/bin or /usr/bin etc.). Package management systems are build on top of this; they additionally resolve 'dependencies', i.e. if a program requires other software (usually libraries) to run, these can also be automatically installed.

GNU make tutorial

GNU make is a CLI build tool, i.e. it helps to easily build and rebuild the code in your project as needed. The make commands should normally be written in a extensionless file named "makefile" in the project folder. Building is then done by just running the shell command "make" within the same folder. What differs makefile from other script files is that they check dependencies before building. For example, let us compare between a makefile and a shell script in the following situation. We have a Fortran script that calculates a function f(x) in different x values and saves the results to a dat file. Then, we plot the results using a python script. We already run the programs and have a plot file. Now we edit the plot file to improve the plot. Using a shell script that compiles the Fortran script and runs the python file would calculate the function values again, which is actually not needed and may waste much time. Using a makefile, we can easily avoid this problem. In fact, it automatically finds out that only the python script was edited, that is, make just runs it again. This makefile may look like: 

fig.png : code.exe plot.py
	.\code.exe
	python plot.py
code.exe : code.f90
	gfortran -Wall -Wextra -o code.exe code.f90
clean : 
	del *.exe *.png *.dat

One should note that this makefile also allows us to easily remove all files with the extensions .exe (binaries), .png (pictures) and .dat (datafiles) just by running "make clean". In fact, clean is here a target that no other target depends on, which means it does not happen unless it is specified in the prompt. You can find example python and Fortran files below.

code.f90 
program f
    implicit none
    real,parameter::a=-10,b=10
    integer,parameter::N=10000
    real,parameter::step=(b-a)/(N-1.)
    real,dimension(N)::x,y
    integer::i
    do i=1,N
        x(i)=a+(i-1.)*step
        y(i)=atan(x(i))
    end do
    open(1, file = 'data.dat')
    do i=1,N
        write(1,*) x(i), y(i)
    end do
end program f
plot.py 
import matplotlib.pyplot as plt
import numpy as np
data = np.loadtxt('data.dat')
x = data[:,0]
y = data[:,1]
fig,ax=plt.subplots()
ax.plot(x,y)
ax.set_xlabel("x")
ax.set_ylabel("y")
ax.set_title("f(x)=arctan(x)")
ax.grid()
plt.savefig("fig.png")
plt.close()

The makefile above was used for explanation purposes. But the project makefiles you would normally see look more like the following one. 

#CONFIG_ADD = yes
FC = ifc
FC_FLAGS = -fast
OBJECTS= main.o sub.o
ifdef CONFIG_ADD
 OBJECTS= $(OBJECTS) add.o
endif
my-program : $(OBJECTS)
	$(FC) $(LD_FLAGS) -o $@ $(OBJECTS)
%.o : %.f90
	$(FC) $(FC_FLAGS) -c -o $@ $<
clean :
	rm *.o
clean-all :
	rm *.o my-program

Here, the Intel Fortran compiler (ifc) is used for compilation. The compiler and the flags are specified in the beginning of the file. This makefile builds the final executable "my-program", and the objects needed for it are also listed. IF CONFIG_ADD is set to yes, add.o is added to the objects.

The next makefile is a one that you could use to compile and run any simple Fortran 90 program using gfortran without any editing. 

FC = gfortran
FC_FLAGS = -Wall -Wextra -g -O0 -fbacktrace
%.exe : %.f90 
	$(FC) $(FC_FLAGS) -o $@ $<
	$@
clean:
	del *.exe
.PHONY: clean

Now, let us discuss how to compile and run a Fortran 90 file called code.f90. %.f90 is used to replace any file in the same directory as the makefile that has the extension .f90. If such a file exists, there will be a target with the same name but with the extension .exe instead. This is called a pattern rule. In our case, the target will be code.exe.

Makefiles always start by making the prerequisites of the first target listed before making that target. Still, running make won't do anything, as using a pattern rule is not enough for making a target, the target should be explicitly specified. Thus, one solution would be to add a target called "all" and add code.exe as a prerequisite for it. This, however, requires editing the makefile, which can be time consuming when many different Fortran files have to be compiled.

The solution recommended here is to just run make code.exe. This way, the target is specified on the fly and it is recognized by the makefile since it matches the wildcard. This is much simpler and requires less editing. One should also mention, that this is only possible because the task itself is not complicated, as we only have to compile a simple Fortran program. Sometimes one needs to compile multiple sources, which can't be done with this makefile. You could find help on this below in the tutorial made by NeuralNine.

The compilation command includes automatic variables, i.e. is variables that don't have to be defined beforehand. $@ is filled with the target. $< is filled with the prerequisites. After the compilation, the executable is run with $@.

Last but not least, we use the target .PHONY to specify the phony targets, i.e the fake ones that are not actual files. Indeed, there could be a file called clean in the same directory, which would confuse the makefile. Using .Phony makes sure that this problem doesn't happen. If one uses the first solution mentioned above, all should also be added to the phony targets.
