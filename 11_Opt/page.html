Optimization

Optimization is a complicated topic, as 'premature optimization is the root of all evil' (D. Knuth). While initially writing code you should focus on good programming rather than on optimization. Additionally, simple optimizations (of loop structures etc.) are usually done more satisfactory by the compiler. The greatest potential for optimization is most often at the algorithmic level, i.e. in the way people have 'translated' their scientific insight into a computing recipe - and this is something the compiler knows nothing about. To find the performance bottleneck of your code, profilers are available, which show you how much time your code spends in which routines. Obviously, it is not worth your time to optimize something, in which your code spends only seconds. Even zeroing the computation time of a part which takes half of the overall computation time will not make your code more than 2 times faster. For typical scientific applications you really have to focus on not more than a few, absolutely central routines. So make sure you write good and not necessarily fast code in the other parts. 'Good' programming style:

There are few criteria for 'good' which are not questioned by one or the other developer, but I think it is fair to say that some general rules can be found, although these of course still depend on the purpose you are coding for. In science, your code should normally be accurate and fast. To be robust and maintainable starts to play a higher role in science, when the size of the project increases. This means the code has to be accessible (intellectually), extendable and testable. Probably the most important strategy to achieve these goals is to 'modularize' your code: put separate functionality in separate files, modules, libraries etc... and make use of this by resorting to existing libraries etc. where possible. Code generators (available or self-written) can be a very good option to generate very complex scientific code reliably. Making use of build systems, version control, and testing is also part of good programming style. Concerning the code itself, most important is a clear structure, with meaningful names (there is no real time to save here with tmp1 to tmp10 ...) and helpful comments, which document interfaces and reasons, not implementations ('this is a loop'). If you have the feeling that you have to explain a lot about a certain piece of code, it is often recommended to instead 'refactor' it, which means to break it down into simpler units, each requiring only a few lines of comments. Finally, remember that 'any code of your own that you haven't looked at for six or more months might as well have been written by someone else' (Eagleson's law), so really make sure you include enough documentation for your future self.
