It turns out that a lot of time within software projects is spent on finding and fixing bugs, i.e. small, but obstructive errors in the source code. At least for larger programming projects, the testing and optimization of code becomes a task on it's own.

Debugging, Testing & Optimization:

Debugging can be (and very often is) as simple as putting an output statement at the place in your code, where things seem to go wrong. If (lots of) errors turn up upon compilation, it is often helpful to just fix the first error, as the other errors might just be follow-up ones. If you have no idea what (or actually where) the problem is, 'delta debugging' might help: put a statement in the middle of your code and see whether the problem occurs before or after. Then again 'half' the problem again and again until you narrowed it down to a few lines. On the command line, but usually more comfortable as part of GUI editors or IDEs, proper debugging programs are available, which let you step line by line through your code during execution while at the same time showing you the values of the variables involved. More complex problems can often be found more easily this way, as you spot 'weird' values for your variables ('not-a-number' etc.) early on.

Debugging (and profiling) tutorial: https://missing.csail.mit.edu/2020/debugging-profiling/

Professional software testing is a business on it's own, but you can easily integrate some ideas into your code development process. At the basic level, languages like Python and C/C++ (but not Fortran -- see why commercial software developers hate it?) allow you to use 'assertions' in your code, where you can make your code stop as soon as things develop in an unexpected direction. The next step up is unit testing, i.e. to write tests with known in- and output for 'units' of your code. For languages like Python and C/C++, unit testing frameworks are available to speed up this process. A good strategy is to also turn all 'complex' bugs which you find later on into tests to avoid running into them again. Another step up is integration testing, i.e. checking whether units are working together correctly and then system testing, where you can for instance include a test suite with example cases and known results as part of your code directory. (People of course have invented many more intermediate levels of testing ... ) A very interesting option is contineous integration, which means that after each git push, a whole range of tests (unit or up to system level) is run automatically. Some people advocate 'test-driven development', i.e. to first make specifications (what should the program or part do?), then write tests based on these specifications and make them properly fail, and only then start to write solutions that will not trigger the tests.

Optimization is a complicated topic, as 'premature optimization is the root of all evil' (D. Knuth). While initially writing code you should focus on good programming rather than on optimization. Additionally, simple optimizations (of loop structures etc.) are usally done more satisfactory by the compiler. The greatest potential for optimization is most often at the algorithmic level, i.e. in the way people have 'translated' their scientific insight into a computing recipe - and this is something the compiler knows nothing about. To find the performance bottleneck of your code, profilers are available, which show you how much time your code spends in which routines. Obivously, it is not worth your time to optimize something, in which your code spends only seconds. Even zeroing the computation time of a part which takes half of the overall computation time will not make your code more than 2 times faster. For typical scientific applications you really have to focus on not more than a few, absolutely central routines. So make sure you write good and not necessarily fast code in the other parts.  'Good' programming style:

There are few criteria for 'good' which are not questioned by one or the other developer, but I think it is fair to say that some general rules can be found, although these of course still depend on the purpuse you are coding for. In science, your code should normally be first of all accurate and fast. To be robust and maintainable starts to play a higher role in science, when the size of the project increases. This means the code has to be accessible (intellectually), extendable and testable. Probably the most important strategy to achieve these goals is to 'modularize' your code: put separate functionality in separate files, modules, libraries etc.. and make use of this by resorting to existing libraries etc. where possible. Code generators (available or self-written) can be a very good option to generate very complex scientific code in a reliable fashion. Making use of build systems, version control, and testing is also part of good programming style. Concerning the code itself, most important is a clear structure, with meaningful names (there is no real time to save here with tmp1 to tmp10 ...) and helpful comments, which document interfaces and reasons, not implementations ('this is a loop'). If you have the feeling that you have to explain a lot about a certain piece of code, it is often recommended to instead 'refactor' it, which means to break it down into simpler units, each requiring only a few lines of comments. Finally, remember that 'any code of your own that you haven't looked at for six or more months might as well have been written by someone else' (Eagleson's law), so really make sure you include enough documentation for your future self.
