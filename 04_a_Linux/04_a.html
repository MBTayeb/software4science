A lot of work in RSEng (and arguably most of it in science-related RSEng) is done under Linux and making use of command line interfaces (CLIs). You should therefore try to get familiar with both before starting to work on the following sections.

Some notes on Linux:

Linux is an open source operating system (OS) alternative to DOS/Windows and MacOS. It has pretty much the same look-and-feel as Windows nowadays, if you like. Unlike Windows or MacOS there is not one current version of Linux but a whole range of ' distributions ', which has it's own advantages and disadvantages . Some distributions are community-driven but commercially backed , i.e. supported by IT companies (the name of the company is put in parenthesis), like Fedora (Red Hat), openSUSE (SUSE), Ubuntu (Canonical Ltd.) and they offer pretty much all the functionality of Windows, plus access to the world of Linux via package management systems that allow you to install new software tools with one line in your terminal. Some distributions are entirely community-driven , like Debian, Slackware, Gentoo, or Arch Linux and they oftentimes offer just a basic system and a package management system. In addition, there are many special-purpose OSs like Android, which are based on Linux. While the desktop consumer market is clearly dominated by Windows, a lot of 'serious' (and loads of not-so-serious) IT-stuff beyond this is ususally done under UNIX/Linux, especially in the area of embedded and/or scientific computing and first of all high-performance computing (HPC). A core reason for this is that the open source approach is self-enhancing: evermore useful tools are developed on the basis of easily accessible existing tools.

A list of Linux distributions can be found on Wikipedia: https://en.wikipedia.org/wiki/List_of_Linux_distributions
(I would recommend to choose on of the more commonly used distributions like Suse, see https://www.opensuse.org/ , or Ubuntu, see https://ubuntu.com/ .)

There are many ways how to run Linux and Windows: The most obvious one is to; get a second machine. The probably less obvious one is that; you can install more than one operating system on a machine, e.g. as dual boot for Linux and Windows. But as nowadays virtualization technology has become quite good, it is usually easier to run either Linux or Windows in a virtual machine on either Windows or Linux, or maybe on a distant (e.g. WWU-IT or IVV NWZ) server. A very nice option for NWZ Windows machines is the installation of the Windows subsystem for Linux (WSL2), which allows running certain Linux distributions within a subsystem of your NWZ Windows (you will have to ask your group Admin for the installation):

https://docs.microsoft.com/de-de/windows/wsl/install-win10
(Currently, graphics are not availabe, but GUI access is upcoming. A nice addition is WindowsTerminal, which allows you to have PowerShell and Bash terminals side by side.)

Some notes on CLIs, Bash and VI:

While you can use modern Linux systems like Windows via graphical user interfaces (GUIs), the real strength of Linux is the fact that really everything can be done via command-line-interfaces (CLIs), which is usually a terminal, where you can type Bash ('Bourne-again shell') commands to modify data, processes etc. Why would you want to do that? First, you can avoid GUIs, which saves resources and is possible also on (even remote) computers where no graphical output is available. But it is even more important that , you can put your commands into scripts (small programs), to automate the work you are doing and to build up a tool box of scripts using simpler scripts, and so an, based on the basic Bash commands. This philosophy of having small programs for specific tasks and solving more complex problems by 'piping' (handing over) data from one to the other is the core of Bash scripting. You should always strive to write scripts instead of doing things by hand, to increase the reproducibility of your work: if you make errors by hand, they are often basically impossible to find, but if your script contains an error, the problem is much easier to spot, as it occurs in a regular fashion. Also, if things go wrong, with a script you can simply delete the whole lot and easily set up directories, files etc. again (and again, ...).  BTW: Windows offers PowerShell for the same purpose, but unlike Linux, where you have a simple shell at the bottom and would (according to the tool box philosophy) turn to a proper scripting language like Python for more complex tasks, PowerShell is a somewhat unfortunate mix-up of a basic shell and a more powerful scripting language. (Feel free to disagree, of course!)

To feel at home on a Linux system you will have to know a certain set of shell commands, so that you can move to and from directories and manipulate files etc. Every Bash course (including the ones listed below) will guide you through this set of commands.

Here is a list of what you have to know: ls, cd, cp, mv, rm, md, rmdir, mkdir, echo

Here is a list of what you should now: cat, cut, paste, head, tail, sort, uniq, wc, seq, tr, bc, man, chmod, diff, find, tar, rsync, ssh, scptop, ps, kill, df, du, hostname, date, pwd, ...

And this is a list of things you can never learn enough about: grep (textsuche), sed (zeileneditor), gawk (textprozessor), ...

In addition, you will have to learn how to assign variables, and how to do loops (do this so and so often ...) and how to use conditions (do this if ...) in your bash scripts.

Another central tool is the editor Vi , because work in a shell usually means the editing of files and Vi is the editor you will find on literally every machine. Vi has a marvelous list of functions, but if you don't like Vi, I would still recommend learning how to use it to just edit files (which is easy, so don't worry).

A very useful tool is SSH ('secure shell') which allows you to access remote computers from your shell via a cryptographic network protocol. It works as easy as 'ssh user@remotecomputer' and afterwards whatever you type will be executed on the distant machine. NWZ computers usually do not allow direct SSH access from outside WWU, but you can first hop to the WWU-IT ssh gateway (e.g. zivlts3.uni-muenster.de ) and then make another hop to the machine you want to access (it still has to allow for SSH access).

To learn more about specific commands you can either: 1. call the command with the -h or --help option (good to look up certain options you always forget again), 2. type info <command> or man <command> (good to get a basic idea of the usage), or directly google whatever you are interested in, e.g. "bash echo" etc.
